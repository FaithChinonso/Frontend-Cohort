Chapter 01

What HTML actually is

If a webpage were a person, HTML would be the skeleton — the structure under everything you see.

HTML stands for HyperText Markup Language. It's not really a "programming" language — you don't write logic or do math in it. You describe what's on a page: here is a heading, here is a paragraph, here is an image. The browser reads your description and draws the result.

Every website on the internet — Google, YouTube, your bank's homepage — starts as HTML. The colors, fonts, and layout you see come from CSS. The interactivity (buttons that respond, forms that validate) comes from JavaScript. But the structure underneath? Always HTML.

HTML is to a webpage what the script is to a movie. Not the costumes, not the lighting — the words that tell the story.

How a browser reads HTML

When you save a file ending in .html and open it in a browser, the browser reads it top to bottom, builds a model of the page in memory (the DOM — Document Object Model), and paints it on screen. Change the file, refresh the browser, see the new version. That's the entire feedback loop.

Quick mental model

HTML is the noun. CSS is the adjective. JavaScript is the verb. "Show a red button that calls my friend." Red = CSS, button = HTML, calls = JavaScript.

You need almost nothing to start

  • A code editor — VS Code is free and what professionals use
  • A web browser — Chrome, Edge, Firefox, Safari all work
  • A folder anywhere on your computer to save your files in

No installation of HTML itself. The browser already speaks it.

Chapter 02

Tags, elements, and attributes

Three words that get used almost interchangeably — but each one means something slightly different.

A tag is a word in angle brackets

<p>Hello world</p>

That whole thing — opening tag, content, closing tag — is an element. The element's name is p (for paragraph). Most elements come in pairs: an opening tag <p> and a closing tag </p> with content in the middle.

A few tags close themselves

Some elements have no content — they exist as one tag. The classic example:

<img src="cat.jpg" alt="A sleepy cat">

You'll also see them written as <img ... /> with a slash before the closing bracket. Both are valid. Other self-closing tags: <br>, <hr>, <input>, <link>, <meta>.

Attributes — extra info on a tag

In the image above, src and alt are attributes. They go inside the opening tag and follow this pattern:

name="value"

You'll meet dozens of attributes, but they all follow that shape. The most important ones early on:

AttributeUsed onWhat it does
href<a>The URL the link points to
src<img>, <script>Source — where the file lives
alt<img>Text shown if the image fails (and read aloud by screen readers)
idAny tagA unique label for that one element
classAny tagA reusable label, used by CSS and JS
titleAny tagTooltip text on hover

Nesting — tags inside tags

Tags wrap around each other. The only rule is: close them in reverse order of opening.

<!-- Right -->
<p>I love <strong>chocolate cake</strong> a lot.</p>

<!-- Wrong -->
<p>I love <strong>chocolate</p> cake</strong>

Browsers are forgiving — they'll usually render the wrong version anyway — but messy nesting causes bugs that take hours to find. Always close tags in the right order.

A note on quotes

Attribute values can use "double" or 'single' quotes — pick one and stick with it. Double quotes are conventional.

Comments — notes the browser ignores

<!-- This is a comment. The browser ignores it. -->
<!-- Use them to leave yourself notes or hide code temporarily. -->

Chapter 03

The page skeleton

Every HTML file you ever write starts with the same handful of lines. Type them once, memorize them, move on.

The starter template

<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>My Page</title>
  </head>
  <body>
    <!-- Everything you see on the page goes here -->
  </body>
</html>

What each line means

LineWhy it's there
<!DOCTYPE html>Tells the browser "this is modern HTML." Always first. Always.
<html lang="en">Wraps the whole document. lang helps screen readers and search engines.
<head>Info about the page — not shown on the page itself.
<meta charset="UTF-8">Lets your page use any character (emoji, accents, Chinese, anything).
<meta name="viewport">Tells phones to render the page at the right size. Without it, mobile sites look tiny.
<title>The text in the browser tab and in Google search results.
<body>Everything the user actually sees lives here.

Hack — Emmet's "!" shortcut

In VS Code, type a single ! on an empty line and press Tab. The whole skeleton above appears instantly. This works in any HTML file — try it now.

The save → refresh loop

  1. Edit the file in VS Code
  2. Save (Ctrl+S on Windows, Cmd+S on Mac)
  3. Switch to the browser and refresh (F5 or Ctrl+R)

That's the entire web-development workflow at this stage. You'll do this thousands of times.

— end of chapter three —

Chapter 04

The everyday tags

If you only learn ten tags, learn these. Together they cover 80% of what you'll ever write.

Headings — <h1> through <h6>

<h1>The most important heading on the page</h1>
<h2>A section heading</h2>
<h3>A sub-section</h3>
<h4>Smaller still</h4>

h1 is the biggest, h6 the smallest. Use only one <h1> per page — usually the page title. Don't pick a heading level for its size (CSS will handle that) — pick it for its importance in the outline.

Paragraphs — <p>

<p>This is one paragraph of text.</p>
<p>And here is another. Each paragraph appears on its own line.</p>

The browser ignores whitespace inside HTML — extra spaces, line breaks, and tabs all collapse to a single space. That's why every paragraph needs its own <p> wrapper.

Links — <a>

<a href="https://google.com">Go to Google</a>
<a href="about.html">About this site</a>
<a href="mailto:hello@me.com">Email me</a>
<a href="#about">Jump to the About section</a>

Four kinds of href values you'll use constantly:

  • Full URL — https://... — go to another site
  • Relative path — about.html — another file in the same folder
  • Mail link — mailto:... — opens the user's email client
  • Anchor — #id — scroll to an element with that id

Security hack — external links

When you open a link in a new tab with target="_blank", always add rel="noopener noreferrer":

<a href="https://google.com" target="_blank" rel="noopener noreferrer">Google</a>

Without it, the linked page can manipulate yours via JavaScript. Modern browsers default to safe behavior, but adding it is the professional habit.

Images — <img>

<img src="pasta.jpg" alt="A bowl of carbonara" width="400">
  • src — path to the image file (relative or full URL)
  • alt — describes the image for blind users and shows if loading fails. Never skip this.
  • width / height — optional, in pixels. Setting them prevents the page from "jumping" as the image loads.

Bold and italic — <strong> and <em>

<p>This is <strong>very important</strong> and this is <em>emphasized</em>.</p>

<strong> renders as bold. <em> renders as italic. You'll also see <b> and <i> — they look the same, but they mean "just style this" instead of "this carries extra meaning." Prefer strong/em for content; screen readers emphasize them when reading aloud.

Line break — <br>

<p>Roses are red<br>Violets are blue</p>

Use sparingly — usually a new paragraph is better. <br> is right for poetry, mailing addresses, song lyrics. Don't use it to add visual space between paragraphs — that's CSS's job.

Horizontal rule — <hr>

A thematic break in the content — like the line at the end of a scene in a novel.

<p>End of part one.</p>
<hr>
<p>Beginning of part two.</p>

Chapter 05

Lists — bullets and numbers

When you have a list of things, use a list. It's a common beginner mistake to use a stack of paragraphs instead.

Unordered list — <ul>

Bulleted. Order doesn't matter. Shopping lists, tags, navigation menus.

<ul>
  <li>Tomatoes</li>
  <li>Basil</li>
  <li>Olive oil</li>
</ul>

Ordered list — <ol>

Numbered. Order matters. Cooking steps, instructions, rankings.

<ol>
  <li>Boil water</li>
  <li>Add salt</li>
  <li>Drop in the pasta</li>
</ol>

You don't type the numbers yourself. The browser numbers them automatically. Reorder the list and the numbers re-number.

Nested lists

A list inside a list. Useful for sub-points.

<ul>
  <li>Fruits
    <ul>
      <li>Apple</li>
      <li>Banana</li>
    </ul>
  </li>
  <li>Vegetables</li>
</ul>

Description lists — <dl>

A lesser-known list for term / definition pairs. Perfect for glossaries, FAQs, recipe metadata.

<dl>
  <dt>Al dente</dt>
  <dd>Pasta cooked so it's still firm to the bite.</dd>

  <dt>Mise en place</dt>
  <dd>Having all your ingredients prepped before you start.</dd>
</dl>

When to use which list

Order matters? <ol>. Just a collection? <ul>. Pairs of term + meaning? <dl>. Don't fake a list with paragraphs — search engines and screen readers can tell the difference.

— end of chapter five —

Chapter 06

Semantic HTML

"Semantic" means "having meaning." Choose tags that describe what something IS, not just how it should look.

Two pages, same look — different meaning

<!-- Bad — every box is a generic div -->
<div>The top of the page</div>
<div>Main content</div>
<div>The footer</div>

<!-- Good — tags carry meaning -->
<header>The top of the page</header>
<main>Main content</main>
<footer>The footer</footer>

Visually identical. But the second version tells the browser, screen readers, and search engines what each section is for. That matters because:

  • Blind users navigating by screen reader can jump between sections ("go to main content")
  • Google ranks pages with clear structure higher
  • Your future self reads the code faster

The semantic tags to know

TagMeaning
<header>Top of a page or section — logo, page title, nav
<nav>A group of navigation links
<main>The main unique content of the page. One per page.
<section>A themed group of content (a chapter, a recipe section)
<article>A self-contained piece — a blog post, a recipe, a tweet
<aside>Tangentially related content — sidebar, pull quote
<footer>Bottom of a page or section — copyright, contact, social links
<figure> + <figcaption>An image (or other content) with a caption
<time>A date or time, machine-readable via the datetime attribute

A real page outline

<body>
  <header>
    <h1>Tasty Bites</h1>
    <nav>
      <a href="/">Home</a>
      <a href="/recipes">Recipes</a>
    </nav>
  </header>

  <main>
    <article>
      <h2>Carbonara</h2>
      <section><h3>Ingredients</h3> ... </section>
      <section><h3>Steps</h3> ... </section>
    </article>
  </main>

  <footer>
    <p>&copy; 2026 Tasty Bites</p>
  </footer>
</body>
If a tag exists for what you're describing, use it. div is what you reach for when nothing else fits.
— end of chapter six —

Chapter 07

div, span, and friends

Sometimes you need a wrapper that has no meaning — just a box to group things or style. That's what div and span are for.

<div> — a generic block

A "block" element takes up the full width of its container and starts on a new line. <div> wraps related stuff together so CSS or JS can target it.

<div class="recipe-card">
  <img src="pasta.jpg" alt="Carbonara">
  <h3>Carbonara</h3>
</div>

<span> — a generic inline piece

An "inline" element only takes up as much space as its content and sits within a line of text. Use <span> to highlight a piece of text without changing its flow.

<p>Price: <span class="price">$12.99</span></p>

Block vs inline — the mental model

┌──────────────────────────────────────────────────┐ │ BLOCK element (h1, p, div, section, ...) │ │ ── takes a full line │ └──────────────────────────────────────────────────┘ ┌──────────────────────────────────────────────────┐ │ Another BLOCK element │ └──────────────────────────────────────────────────┘ This is a paragraph with an [INLINE span] inside it flowing along with the regular text.

Rule of thumb

Reach for a semantic tag first. If none fits, reach for <div> (block) or <span> (inline). Never use them when something more meaningful would do.

— end of chapter seven —

Chapter 08

Forms — letting users type

Whenever a user types, picks, or clicks something to send data back — that's a form. Logins, searches, signups, contact pages, comment boxes. All the same building blocks.

The minimum form

<form>
  <label for="email">Email address</label>
  <input type="email" id="email" name="email">

  <button type="submit">Sign up</button>
</form>

Three pieces to know: input, label, button.

Input types — set with type="..."

TypeWhat you get
textPlain single-line text (default)
emailEmail-shaped text. Phones show @ on the keyboard.
passwordHides characters as dots
numberNumbers only, with up/down arrows
telPhone numbers — shows the number keypad on mobile
urlValidates as a URL
dateNative date picker calendar
timeTime picker
colorNative color picker
rangeA slider
fileFile upload dialog
checkboxA tickbox
radioPick one of many (group with the same name)
searchSearch box (with a clear "x" in some browsers)
submitA submit button

Why labels matter

The <label> tag connects descriptive text to an input. Connect them by matching the label's for attribute to the input's id.

<label for="username">Username</label>
<input type="text" id="username" name="username">

Why bother?

  • Clicking the label focuses the input — bigger target, less friction
  • Screen readers read the label when reading the input
  • SEO and accessibility audits reward you

Useful input attributes

AttributeWhat it does
placeholderGrey hint text shown until the user types
requiredForm won't submit unless this is filled
valueDefault value
maxlength / minlengthCharacter limits
min / max / stepFor number, date, range
patternA regex the value must match
autocompleteTell browsers what to autofill: email, name, tel, current-password
autofocusFocus this field automatically on page load
disabledGreyed out, can't be edited or submitted
readonlyVisible value, not editable, still submitted

Other form controls

Multi-line text — <textarea>

<label for="msg">Your message</label>
<textarea id="msg" name="message" rows="5"></textarea>

Dropdown — <select>

<label for="meal">Favorite meal</label>
<select id="meal" name="meal">
  <option value="breakfast">Breakfast</option>
  <option value="lunch" selected>Lunch</option>
  <option value="dinner">Dinner</option>
</select>

Checkboxes and radios

<!-- Checkboxes — pick zero or many -->
<label><input type="checkbox" name="diet" value="vegan"> Vegan</label>
<label><input type="checkbox" name="diet" value="gluten-free"> Gluten-free</label>

<!-- Radios — pick exactly one. Same name = same group. -->
<label><input type="radio" name="size" value="small"> Small</label>
<label><input type="radio" name="size" value="large"> Large</label>

Trick — wrap inputs in labels

For checkboxes and radios, putting the <input> inside the <label> means you can skip for/id entirely. Clicking the label still focuses the input.

Group related fields — <fieldset>

<fieldset>
  <legend>Size</legend>
  <label><input type="radio" name="size"> Small</label>
  <label><input type="radio" name="size"> Medium</label>
  <label><input type="radio" name="size"> Large</label>
</fieldset>

<fieldset> draws a box around related fields. <legend> labels the group.

Buttons

<button type="submit">Send</button>
<button type="reset">Clear</button>
<button type="button">Do something</button>

Three button types. Inside a form, the default is submit — so always set type="button" on any button that shouldn't submit the form (otherwise Enter in any input will trigger it).

— end of chapter eight —

Chapter 09

Tables — for actual tabular data

Tables are for data in rows and columns. Not for layout. Layout uses CSS Grid and Flexbox.

The shape of a table

<table>
  <thead>
    <tr>
      <th>Recipe</th>
      <th>Time</th>
      <th>Category</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Carbonara</td>
      <td>25 min</td>
      <td>Dinner</td>
    </tr>
  </tbody>
</table>
TagWhat it is
<table>The outer wrapper
<thead>The header row group
<tbody>The data row group
<tr>Table row
<th>Header cell (bold + centered by default)
<td>Data cell
<caption>Optional title for the table

Don't use tables for layout

Years ago, designers built whole webpage layouts using nested tables. Don't. Tables tell screen readers "this is tabular data" — which confuses everything when it's actually a sidebar. Use CSS for layout, tables for data.

— end of chapter nine —

Chapter 10

Multi-page sites

Real websites have more than one page. You make them by creating more HTML files and linking them.

One folder, several files

my-site/ ├── index.html ← the homepage (always named "index.html") ├── about.html ├── contact.html └── images/ ├── logo.png └── hero.jpg

Use <a> to link between them:

<a href="about.html">About us</a>
<a href="images/logo.png">View our logo</a>

Why index.html is special

When someone visits a folder URL (like mysite.com/ or mysite.com/recipes/), the server looks for a file called index.html and serves it. That's why your homepage is always named index.html.

Paths — relative vs absolute

PathWhat it means
about.htmlSame folder as the current file
recipes/carbonara.htmlSubfolder named recipes
../about.htmlOne folder UP, then about.html. .. means "parent folder."
/about.html"Root" of the site — starts at the top no matter where you are
https://example.com/An absolute URL — a full address to anywhere on the web
— end of chapter ten —

Chapter 11

Tips & little hacks

Small habits that separate clean HTML from messy HTML. Steal these and your code will feel professional from day one.

Use Emmet shortcuts in VS Code

VS Code has built-in Emmet — type a tag name and press Tab:

  • p → <p></p>
  • ul>li*3 → an unordered list with 3 list items
  • nav>a*4 → a nav with 4 anchor tags
  • .recipe-card → <div class="recipe-card"></div>
  • #header → <div id="header"></div>
  • ! → the full HTML5 skeleton
  • lorem20 → 20 words of placeholder Latin

Always set the language

The lang attribute on <html> tells screen readers what language to pronounce — and tells search engines what language the page is.

<html lang="en">
<html lang="en-US">
<html lang="es">

Always include the viewport meta tag

<meta name="viewport" content="width=device-width, initial-scale=1">

Without it, mobile browsers render your page at desktop width and zoom out — making everything tiny.

Always write alt text

For meaningful images: describe what's in it.

For decorative images (icons, dividers): use empty alt="" so screen readers skip them.

<!-- Meaningful -->
<img src="chef.jpg" alt="Chef Anna mixing pasta">

<!-- Decorative -->
<img src="sparkle.svg" alt="">

HTML entities — write special characters

Some characters can't be typed directly because the browser would think they're HTML. Use entities — they start with & and end with ;:

EntityRenders asUse for
&amp;&The literal ampersand
&lt; / &gt;< >Less than / greater than
&copy;©Copyright
&trade;™Trademark
&reg;®Registered
&nbsp; A non-breaking space (text won't wrap here)
&mdash;—An em dash
&ndash;–An en dash
&hellip;…Ellipsis

Tip: emoji work natively now — you can just paste them. <p>Hello 🎉</p> works fine.

Make phone numbers and emails tappable

<a href="tel:+1234567890">Call us</a>
<a href="mailto:hello@me.com?subject=Hi">Email</a>
<a href="sms:+1234567890">Text us</a>

On phones, these open the dialer, mail app, and messages app respectively.

Force a download instead of opening

<a href="recipe.pdf" download>Download recipe</a>
<a href="recipe.pdf" download="carbonara-recipe.pdf">Download</a>

The download attribute makes the browser save instead of open. Add a value to rename the file on save.

Data attributes — stash data on any tag

Any attribute starting with data- is your own custom data. JavaScript can read it back. CSS can use it for styling.

<button data-category="dinner" data-recipe-id="42">
  Carbonara
</button>

Native expand/collapse — <details>

<details>
  <summary>Click to see the recipe</summary>
  <p>Boil pasta. Mix eggs and cheese. Combine.</p>
</details>

A toggleable section that works without any JavaScript. Great for FAQs, optional details, mobile menus.

The <dialog> element — native modals

<dialog id="my-modal">
  <p>Are you sure?</p>
  <button onclick="this.closest('dialog').close()">Cancel</button>
</dialog>

A real dialog box, focus-trapped, ESC closes it, all without JS libraries.

Mark important text — <mark>

<p>Don't skip the <mark>resting time</mark> — it matters.</p>

Renders with a highlighter-like background. Good for search results too.

Show keyboard shortcuts — <kbd>

<p>Press <kbd>Ctrl</kbd> + <kbd>S</kbd> to save.</p>

Group form fields visually

Wrap label + input pairs in a <div> so CSS can space them out:

<div class="field">
  <label for="email">Email</label>
  <input type="email" id="email">
</div>

autocomplete saves users typing

<input type="text"     autocomplete="name">
<input type="email"    autocomplete="email">
<input type="tel"      autocomplete="tel">
<input type="password" autocomplete="current-password">

The browser pre-fills these from saved info. Users love this — they won't even know your form is well-written, just that filling it is fast.

Use semantic time

Published <time datetime="2026-06-15">June 15, 2026</time>

The datetime attribute is machine-readable. Screen readers, search engines, and calendars all understand it.

The <picture> element — different images for different screens

<picture>
  <source media="(max-width: 600px)" srcset="small.jpg">
  <source media="(max-width: 1200px)" srcset="medium.jpg">
  <img src="large.jpg" alt="Pasta">
</picture>

Phone users download small.jpg, desktop users download large.jpg. Saves bandwidth.

Loading lazy on images below the fold

<img src="recipe.jpg" alt="..." loading="lazy">

The browser only loads the image when the user scrolls near it. Free performance win.

Chapter 12

Common mistakes

Every beginner makes these. Spotting them early saves hours of confusion.

1. Forgetting to close tags

<!-- Wrong -->
<p>This paragraph never ends
<p>So this one joins it weirdly</p>

2. Using headings for size, not meaning

Don't write <h3> just because you want smaller text. Pick the heading level based on outline depth. Use CSS for size.

3. Multiple <h1>s on one page

One per page. The browser allows more, but it hurts SEO and accessibility.

4. Skipping alt text

The most common accessibility failure. alt="" for decoration is fine. Just don't leave it off entirely.

5. <br> as spacing

<!-- Bad -->
<p>Paragraph one.</p>
<br><br>
<p>Paragraph two.</p>

Use CSS margin for space between elements. <br> is for line breaks within text only.

6. div for everything

Code that's all <div> is a sign you haven't picked the right semantic tags. Use header, nav, main, article, section, footer when they fit.

7. Inline styles for everything

<!-- Bad -->
<p style="color: red; font-size: 18px;">Hello</p>

Inline styles are hard to change later. Put styling in a separate CSS file. Use class on the HTML.

8. Forgetting the viewport meta tag

Without it, your site looks tiny on phones. Always include it in <head>.

9. Buttons that aren't <button>

If it's clickable and does something, make it a real <button>. Don't use <div onclick="...">. Buttons have keyboard support, focus styles, and screen-reader announcements built in.

10. Reusing the same id

id values must be unique per page. If you need to reuse, use class instead.

— end of chapter twelve —

  4. TypeScript — JavaScript with safety rails. (Lesson 07.)
  5. Tailwind CSS — a faster way to style. (Lesson 08.)

Resources worth bookmarking

  • MDN Web Docs (developer.mozilla.org) — the official, most accurate reference for every HTML tag.
  • HTML5 Doctor — short articles on semantic HTML choices.
  • Can I Use (caniuse.com) — check which browsers support a tag or attribute.
  • The W3C HTML Validator — paste your HTML, get a list of errors.

Build something — anything

Reading about HTML teaches you the vocabulary. Building teaches you to write. Don't wait until you "know enough" — you'll never feel ready. Pick something tiny:

  • A one-page bio about yourself
  • A page that lists your three favorite movies, each with an image
  • A simple recipe page like the one we built in class
  • A contact form for an imaginary business
The fastest way to learn HTML is to build five small, ugly websites in a week. The second-fastest is to build one. Don't read another tutorial first — open VS Code now.
